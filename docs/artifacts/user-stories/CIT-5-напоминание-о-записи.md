## User Story

**Как** клиент, записавшийся к мастеру,  
**я хочу** получать напоминание в Telegram за 2 часа до начала записи,  
**чтобы** не забыть о визите и успеть подготовиться или отменить, если не смогу прийти.

## Acceptance Criteria

### AC-1: Напоминание доставляется вовремя

```gherkin
Given клиент имеет бронирование со статусом confirmed
  And у клиента есть активная Подписка в Telegram
  And до start_at бронирования остаётся ровно 2 часа (±5 минут допуск планировщика)
When планировщик запускает job отправки напоминаний
Then клиент получает Telegram-сообщение с информацией о предстоящей записи
  And сообщение содержит: имя мастера, название услуги, дату и время записи, адрес бизнеса
  And сообщение НЕ отправляется повторно для той же записи
```

### AC-2: Клиент без Telegram-подписки — пропуск без ошибки

```gherkin
Given клиент имеет бронирование со статусом confirmed
  And у клиента НЕТ активной Подписки в Telegram (не привязал бота)
When планировщик запускает job отправки напоминаний
Then напоминание НЕ отправляется
  And система логирует событие: только booking_id, причина="no_subscription"
    (без телефона и имени клиента — C-002)
  And остальные напоминания в батче продолжают обрабатываться
```

### AC-3: Статус бронирования — только confirmed

```gherkin
Given бронирование имеет статус pending, cancelled, completed или rejected
When планировщик проверяет бронирования для отправки напоминаний
Then напоминание для этого бронирования НЕ отправляется
  And статус бронирования НЕ изменяется
```

### AC-4: Соблюдение лимитов Telegram API (C-003)

```gherkin
Given в системе одновременно 100+ бронирований подходят под условие напоминания
When планировщик формирует батч отправки
Then сообщения отправляются через rate limiter: не более 30 сообщений/сек суммарно
  And в один чат — не более 1 сообщения/сек
  And при получении HTTP 429 от Telegram API система применяет exponential backoff
  And все сообщения доставляются (возможна задержка до 5 минут)
```

### AC-5: Идемпотентность — повторная отправка исключена

```gherkin
Given напоминание для booking_id=X уже было отправлено (reminder_sent=true)
When планировщик повторно обрабатывает то же бронирование (например, после сбоя)
Then повторное сообщение клиенту НЕ отправляется
  And система логирует: "reminder already sent for booking_id=X"
```

### AC-6: Бронирование отменено до момента отправки напоминания

```gherkin
Given бронирование было переведено в статус cancelled или rejected
  And напоминание для этого бронирования ещё не было отправлено
When планировщик запускает job
Then напоминание НЕ отправляется
  And событие логируется: booking_id, причина="booking_not_confirmed"
```

## Бизнес-сценарии

### Happy Path

1. Клиент создаёт бронирование → провайдер подтверждает → статус `confirmed`
2. Планировщик (cron/celery beat) каждые N минут выбирает `confirmed`-записи,
   у которых `start_at` ∈ [now+2h, now+2h+интервал_планировщика]
   и флаг `reminder_sent = false`
3. Для каждой записи: находит Подписку по телефону клиента
   (тип `CLIENT_PHONE` или `CLIENT_BOOKING`)
4. Отправляет Telegram-сообщение через rate limiter (C-003)
5. Устанавливает `reminder_sent = true` на бронировании
6. Клиент получает сообщение, приходит вовремя ✅

### Альтернативный путь A: клиент не подписан на бота

- Шаг 3: Подписка не найдена → пропуск, лог `booking_id` (без PII, C-002)
- Напоминание не отправлено, остальной батч продолжается

### Альтернативный путь B: провайдер не подтвердил запись

- Шаг 2: бронирование в статусе `pending` → не попадает в выборку → пропуск

### Edge Case 1: запись создана менее чем за 2 часа до старта

- `start_at` = now + 90 минут → на момент создания уже прошло «окно» напоминания
- Напоминание **НЕ** отправляется
- ⚠️ **OQ-1:** нужно ли в этом случае отправлять напоминание немедленно?

### Edge Case 2: смена часового пояса

- Все бронирования хранятся в UTC. Расчёт 2 часов — от UTC `start_at`
- ⚠️ **OQ-2:** бизнесы в разных часовых поясах Казахстана (+5 / +6)?
  Нужно ли хранить timezone бизнеса?

### Edge Case 3: Telegram-бот недоступен (downtime)

- HTTP 5xx от Telegram API → retry с exponential backoff (C-003)
- После восстановления: если `start_at` уже прошёл → напоминание не отправляется

## Out of Scope

| # | Что НЕ входит в эту историю                                        |
|---|---------------------------------------------------------------------|
| 1 | Настройка интервала напоминания провайдером (1ч / 2ч / 24ч)        |
| 2 | Inline-кнопка отмены записи прямо из напоминания                   |
| 3 | SMS или email как fallback-канал                                    |
| 4 | Напоминание провайдеру/мастеру о предстоящей записи                |
| 5 | Отслеживание доставки (read receipt, delivery status)               |
| 6 | Повторное напоминание (например, за 30 минут до записи)            |
| 7 | Рейтинг мастера после визита (техдолг, термин #21 глоссария)       |
